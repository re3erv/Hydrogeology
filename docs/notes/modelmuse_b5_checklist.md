А верхние пакеты — это границы 3 рода, или head-dependent flux boundaries: они не задают напор в ячейке напрямую, а задают связь между ячейкой модели и внешним объектом через сопротивление / проводимость. В документации MODFLOW RIV и GHB прямо описаны как head-dependent flux boundaries, то есть границы, где поток зависит от напора в ячейке. 
Главная идея такая:
Q=C (Hext−hcell)Q = C\,(H_{ext} - h_{cell})Q=C(Hext−hcell)
где:
Q       — поток между внешним объектом и модельной ячейкой
C       — conductance, то есть гидравлическая связь
H_ext   — внешний уровень: stage, drain elevation или boundary head
h_cell  — рассчитанный напор в ячейке модели
То есть MODFLOW не говорит: “в верхней ячейке напор должен быть 7”.
Он говорит: “рядом есть объект с уровнем 7, и вода будет перетекать туда или оттуда в зависимости от разности напоров”.
Отличие от CHD
CHD — граница 1 рода:
h задан жёстко
поток считается как результат
Например, если слева CHD = 10, то ячейки левого столбца не могут стать 9.8 или 10.2. Они зафиксированы на 10.
RIV / DRN / GHB — границы 3 рода:
h в ячейке считается моделью
поток через границу зависит от h
Поэтому в твоих результатах верхняя средняя ячейка не равна строго 7:

DRN: top_mid_head ≈ 4.39

Схема:

~~~~~~~~~~~~~~

------------

Поэтому RIV похож на GHB, но более “речной”: у него есть уровень воды и дно русла.

cond в реальной модели обычно определяют не как одно “измеряемое поле”, а как эффективный параметр сопротивления связи между ячейкой модели и внешним объектом: рекой, дреной, озером, морем, удалённым водоносным горизонтом.
Для MODFLOW 6 это именно hydraulic conductance: у GHB — проводимость интерфейса между ячейкой водоносного горизонта и boundary, у RIV — проводимость руслового ложа, у DRN — проводимость интерфейса между ячейкой и дреной. 
1. Базовая идея
Почти всегда начинаем с закона Дарси:
C=K AMC = \frac{K\,A}{M}C=MKA
где:
C — conductance, м²/сут
K — hydraulic conductivity материала между boundary и aquifer, м/сут
A — площадь контакта, м²
M — толщина сопротивляющего слоя / путь фильтрации, м
Для линейного объекта, например реки или канавы:
A = L · W
тогда:
C = Kbed · L · W / M
Это же даёт ModelMuse: conductance имеет единицы L²/T и часто записывается как K L W / M, где L — длина объекта в ячейке, W — ширина, M — толщина осадков между boundary и ячейкой. 
2. Для RIV — River Package
Для реки cond обычно считают через русловые отложения:
cond = K_riverbed · L_river_in_cell · W_river / M_riverbed
Например:
K_riverbed = 0.1 м/сут
L = 100 м
W = 10 м
M = 0.5 м
Тогда:
cond = 0.1 · 100 · 10 / 0.5 = 200 м²/сут
Что нужно оценить в поле:
1. уровень воды в реке — stage
2. отметка дна реки — rbot
3. ширина русла — W
4. длина русла в каждой ячейке — L
5. толщина русловых отложений — M
6. K русловых отложений — K_riverbed
stage, cond и rbot — именно три входных параметра RIV в MODFLOW 6. 
На практике самая неопределённая величина — обычно K русловых отложений, особенно если есть заиление, органика, глина или кольматация. Поэтому K_riverbed часто задают зонально: песчаный участок — выше, заиленный участок — ниже.
3. Для GHB — General Head Boundary
Для GHB смысл немного другой. Это не обязательно реальный слой осадков под рекой. Это может быть удалённая область, озеро, море, соседний водоносный горизонт, граница региональной модели.
Базовая оценка:
cond = K_connection · A / L_connection
где:
K_connection — эффективная K между ячейкой и внешней областью
A            — площадь границы ячейки
L_connection — расстояние от ячейки до места, где задан bhead
Например, если GHB стоит на боковой границе модели:
K = 10 м/сут
насыщенная мощность b = 20 м
ширина ячейки вдоль границы = 100 м
расстояние до внешнего постоянного уровня = 1000 м
Тогда площадь контакта:
A = b · width = 20 · 100 = 2000 м²
и:
cond = 10 · 2000 / 1000 = 20 м²/сут
То есть cond = 20 в твоей учебной модели можно физически представить как связь с удалённой областью через сопротивление, например через 1000 м водоносного горизонта с K = 10 м/сут, мощностью 20 м и шириной ячейки 100 м.
Для GHB в MODFLOW 6 задаются bhead и cond, причём bhead — это внешний boundary head, а cond — проводимость интерфейса между ячейкой и boundary. 
4. Для DRN — Drain Package
Для дренажа формально можно начать так же:
cond = K_contact · A_contact / M_contact
Но физически DRN часто труднее оценить напрямую, потому что реальная дренажная канава или труба собирает воду не только через “тонкий слой”, а через сложную область притока.
Для канавы можно грубо использовать:
cond = K_bank_or_bed · L_drain_in_cell · W_effective / M_effective
Для горизонтальной дрены/трубы часто cond становится калибровочным параметром, потому что эффективная площадь притока зависит от глубины дрены, радиуса трубы, фильтровой обсыпки, расстояния до водоносного слоя и сетки модели.
MODFLOW 6 определяет cond в DRN как hydraulic conductance интерфейса между aquifer и drain; вода через DRN уходит, когда напор выше отметки дрены. 
5. Практический алгоритм для реальной модели
Я бы делал так.
Шаг 1. Определить физический объект
река / ручей       → RIV или SFR
озеро / море       → GHB, LAK или CHD, зависит от задачи
канава / родник    → DRN
дренажная труба    → DRN
удалённый водоносный горизонт → GHB
Шаг 2. Нарезать boundary по ячейкам
cond задаётся на каждую boundary-ячейку, а не один раз на всю реку или всю границу.
Для каждой ячейки считаешь:
L_i — длина реки / дрены внутри ячейки
W_i — ширина реки / эффективная ширина контакта
M_i — толщина сопротивляющего слоя
K_i — K этого слоя
И затем:
cond_i = K_i · L_i · W_i / M_i
Шаг 3. Взять начальные K и M из полевых данных
Источники:
- литология русла / берегов / дна канав
- керны, шурфы, буровые описания
- slug tests / pumping tests рядом с рекой
- seepage meter, piezometers near river
- разность уровней река–подземные воды
- измеренный baseflow / seepage gain-loss по участкам реки
Для русла реки особенно важна не K основного водоносного горизонта, а K руслового ложа. Оно может быть на порядки ниже из-за ила и кольматации.
Шаг 4. Проверить через водный баланс
После первого расчёта смотри не только heads, но и budget:
RIV leakage
GHB flow
DRN discharge
Если модель даёт, например, приток из реки 10 000 м³/сут, а по полевым оценкам река теряет только 500 м³/сут, значит cond или stage/rbot завышены.
Шаг 5. Калибровать cond
cond почти всегда калибруют, потому что он содержит много неизвестных:
cond = геометрия + K контакта + толщина сопротивления + масштаб ячейки + упрощение физики
Калибровать лучше не каждую ячейку отдельно, а зонами:
RIV_cond_sandy
RIV_cond_silty
GHB_cond_north_boundary
DRN_cond_main_ditches
DRN_cond_minor_ditches
Цели калибровки:
- уровни в наблюдательных скважинах
- расходы родников
- baseflow рек
- gain/loss участки рек
- дебиты дренажа
- сезонная динамика уровней
6. Очень важное предупреждение про сетку
cond зависит от размера ячейки. Если ты изменила/изменил сетку, старые cond обычно нельзя просто оставить как есть.
Например, если река проходит через ячейку, то при уменьшении размера ячейки длина реки L_i внутри каждой ячейки тоже меняется. ModelMuse прямо предупреждает, что при изменении grid size conductance может потребовать пересчёта или перекалибровки. 
7. Как выбрать первое значение, если данных мало
Для учебной или первой рабочей модели можно так:
RIV:
  cond = K_riverbed · L · W / M

GHB:
  cond = K_aquifer · A_boundary / distance_to_external_head

DRN:
  cond = K_local · L · W_eff / M_eff
  затем калибровать по дренажному расходу
И сразу сделать чувствительность:
cond × 0.1
cond × 1
cond × 10
cond × 100
Если heads и budget почти не меняются — boundary слабо влияет.
Если меняются сильно — cond критический параметр, его надо обосновывать особенно тщательно.
Коротко
cond в реальной модели определяют так:
1. Через физическую формулу C = K·A/M
2. По геометрии объекта внутри конкретной ячейки
3. По K и толщине сопротивляющего слоя
4. Потом обязательно проверяют по water budget
5. И почти всегда калибруют по уровням и расходам
Самое правильное понимание: cond — это не просто табличное значение, а эффективная проводимость контакта между модельной ячейкой и внешним гидрологическим объектом.
что такое flow rate на изображении?
Thought for a second
Flow rate на изображении — это расход воды через выбранную границу или ячейку, то есть объём воды за единицу времени.
Если единицы модели:
длина = метры
время = сутки
то flow rate будет:
м³/сут
В твоей модели
Для пакетов RIV, DRN, GHB flow rate — это не скорость фильтрации, а именно объёмный расход обмена между модельной ячейкой и внешним объектом.
Например для GHB:
bhead = 7
cond = 20
h_cell = 5
тогда:
Q = cond · (bhead - h_cell)
Q = 20 · (7 - 5) = 40
То есть flow rate = 40 м³/сут.
Что значит знак flow rate
Обычно в бюджетах MODFLOW смысл такой:
положительный flow rate  → вода входит в модель
отрицательный flow rate  → вода выходит из модели
Но в разных окнах ModelMuse знак может отображаться с точки зрения пакета или с точки зрения модели, поэтому важно смотреть подпись: IN, OUT, FROM, TO, Rate In, Rate Out.
Для твоих пакетов физически:
GHB:
  flow rate > 0 → внешний резервуар питает модель
  flow rate < 0 → модель отдаёт воду во внешний резервуар

RIV:
  flow rate > 0 → река питает водоносный горизонт
  flow rate < 0 → водоносный горизонт разгружается в реку

DRN:
  почти всегда flow rate ≤ 0 с точки зрения модели
  потому что дренаж только забирает воду
Главное отличие от velocity
Flow rate — это расход:
м³/сут
Velocity / specific discharge — это скорость или удельный расход:
м/сут
То есть flow rate отвечает на вопрос:
сколько воды проходит?
а velocity отвечает на вопрос:
с какой интенсивностью или скоростью движется вода?
В твоей B5-модели flow rate на верхней границе показывает, сколько воды каждая ячейка получает от RIV/GHB или отдаёт в RIV/DRN/GHB.
как в model muse увидеть цифру flow rate?
Нужно различить две вещи: flow rate как бюджет пакета целиком и flow rate по отдельной ячейке/объекту. Я дам оба способа, потому что в ModelMuse они смотрятся в разных местах.
Thought for 25s
В ModelMuse число flow rate можно увидеть тремя способами.
1. Самый простой: общий расход по пакету в listing file
После запуска модели ModelMuse обычно открывает текстовый файл результатов — listing file. В нём в конце есть общий водный баланс. Документация ModelMuse говорит, что после успешного запуска listing file открывается в текстовом редакторе, а в конце каждого time step находится общий water budget. 
Ищи в этом файле строки типа:
RIV
DRN
GHB
CHD
TOTAL IN
TOTAL OUT
Для твоих моделей это даст суммарный расход по всему пакету, например общий поток через все ячейки RIV или GHB.
Это отвечает на вопрос:
Сколько всего воды вошло/вышло через RIV / DRN / GHB?
2. Flow rate по отдельным ячейкам: импортировать .cbc
У тебя это уже включено. В файлах gwf.oc для трёх моделей записано:
SAVE BUDGET ALL
BUDGET FILEOUT mf6_05_riv.cbc
BUDGET FILEOUT mf6_05_drn.cbc
BUDGET FILEOUT mf6_05_ghb.cbc
То есть после запуска должны появиться файлы:
riv/mf6_05_riv.cbc
drn/mf6_05_drn.cbc
ghb/mf6_05_ghb.cbc
ModelMuse умеет импортировать heads, drawdowns и flows через:
File → Import → Model Results...
В документации это окно называется Select Model Results to Import Dialog Box; оно используется для импорта head, drawdown, flows и других результатов, а также может раскрасить или оконтурить сетку по выбранному набору данных. 
Порядок действий:
1. Открой модель RIV / DRN / GHB в ModelMuse.
2. Убедись, что модель уже запущена.
3. File → Import → Model Results...
4. Выбери файл .cbc:
   riv/mf6_05_riv.cbc
   или drn/mf6_05_drn.cbc
   или ghb/mf6_05_ghb.cbc
5. В списке data sets выбери нужный flow term:
   RIV, DRN, GHB или CHD.
6. В Color or contour grid выбери этот dataset.
7. Нажми OK.
После этого ячейки можно раскрасить по расходам. Это уже не общий баланс, а расход по конкретным boundary cells.
3. Если хочешь именно число в конкретной ячейке
После импорта .cbc результата:
1. Выбери импортированный dataset в списке Data Sets.
2. Щёлкни по нужной ячейке.
3. Смотри значение dataset для этой ячейки в окне/таблице значений ячейки.
Название dataset может быть примерно таким:
RIV
DRN
GHB
CHD
FLOW-JA-FACE
Для твоего вопроса нужны именно:
RIV — расход через river boundary
DRN — расход через drain boundary
GHB — расход через general-head boundary
FLOW-JA-FACE — это уже потоки между соседними ячейками модели, не поток через RIV/DRN/GHB.

положительный расход → обычно приток в модель
отрицательный расход → обычно отток из модели
Для твоей B5-модели:

GHB < 0  → модель отдаёт воду во внешний объект

RIV > 0  → река питает водоносный горизонт
RIV < 0  → водоносный горизонт разгружается в реку

DRN обычно ≤ 0 → дренаж забирает воду из модели
